将 main 分支（最终版）的最新改动同步到 yunjin-resume 测试分支（部署在子路径，永不合并回 main），保留测试分支的 “子路径部署” 特性，同时更新已有冲突的 PR #5（无需新建 PR）。

因测试分支不反向合并回 main，优先用 rebase（变基）替代 merge，避免多余的合并提交，保持提交记录整洁。

---

## 一、核心操作流程（适配已有 PR 的场景）
### 1. 先关联本地分支与远程 PR 分支
确保你本地操作的 `yunjin-resume` 分支，就是 PR #5 对应的分支：
```bash
# 1. 切换到 yunjin-resume 分支（PR 源分支）
git checkout yunjin-resume

# 2. 拉取该分支的最新代码（包含 PR 已有的 2 个提交）
git pull origin yunjin-resume

# 3. 拉取 main 分支的最新最终版代码
git fetch origin main
```

### 2. 同步 main 改动到 yunjin 分支（解决冲突）
这里优先用 `rebase`（因为 yunjin 不合并回 main，提交记录更整洁）：
```bash
# 基于最新 main 变基 yunjin 分支
git rebase origin/main
```
执行后 Git 会提示冲突文件（就是 PR 里显示的 keywords.json、projects.ts 等）。

### 3. 解决冲突（按你的测试分支需求）
#### 场景 1：保留 main 最终版内容（放弃 yunjin 的删除）
```bash
# 用 main 版本覆盖冲突文件（--theirs 代表 main 分支）
git checkout --theirs app/keywords.json
git checkout --theirs content/en/projects.ts
git checkout --theirs content/zh-hk/projects.ts
# 把所有冲突文件都执行一遍

# 标记冲突已解决
git add .

# 继续变基流程
git rebase --continue
```

#### 场景 2：保留 yunjin 的删除（仅测试分支不需要这些文件）
```bash
# 确认删除冲突文件（仅 yunjin 分支生效）
git rm app/keywords.json
git rm content/en/projects.ts
git rm content/zh-hk/projects.ts
# 所有冲突文件执行 git rm

# 标记冲突已解决
git add .

# 继续变基流程
git rebase --continue
```

### 4. 推送改动到远程，更新 PR #5
因为变基重写了提交记录，需要用 `--force-with-lease` 推送（仅测试分支可这么做）：
```bash
# 推送更新后的 yunjin 分支到远程
git push origin yunjin-resume --force-with-lease
```
✅ 推送完成后，GitHub 上的 PR #5 会自动更新：
- 冲突提示会消失（如果冲突都解决了）；
- PR 的提交记录会变成“基于最新 main 的线性记录”；
- Vercel 也会重新部署测试分支（保留子路径特性）。

---

## 二、关键注意事项（针对已有 PR）
1. **如果 PR 有其他协作者**：
   因为用了 `--force-with-lease` 推送，要确保其他协作者没有在这个分支上做改动，避免覆盖他们的代码（你的场景是自己的分支，风险极低）。

2. **如果不想用 rebase，只想用 merge**：
   若你更习惯保留 merge 提交（哪怕记录乱一点），可以替换步骤 2：
   ```bash
   # 用 merge 替代 rebase
   git merge origin/main
   # 解决冲突后提交
   git commit -m "Merge main into yunjin-resume (测试分支同步最终版)"
   # 推送（无需 force）
   git push origin yunjin-resume
   ```
   这种方式也会更新 PR #5，只是提交记录会多一条 merge 记录。

3. **验证 PR 更新结果**：
   推送后打开 GitHub 的 PR #5 页面，检查：
   - “This branch has conflicts that must be resolved” 提示消失；
   - “Files changed” 里的改动是你预期的（同步了 main 且保留测试分支特性）；
   - Vercel 部署状态变成 “Ready”。

---

## 三、兜底回滚（如果 PR 更新出错）
如果推送后发现改动不符合预期，想恢复 PR 到之前的状态：
```bash
# 1. 先查看 yunjin 分支的提交记录，找到 PR 变基前的提交哈希
git log --oneline

# 2. 回滚到该提交（替换 <old-commit-hash> 为实际哈希）
git reset --hard <old-commit-hash>

# 3. 强制推送到远程，恢复 PR 状态
git push origin yunjin-resume --force-with-lease
```

### 总结
1. 已有 PR 时，无需新建 PR，只需**本地同步 main 并解决冲突后推送**，PR 会自动更新。
2. 测试分支优先用 `rebase` 同步 main，提交记录更整洁，推送时用 `--force-with-lease` 更安全。
3. 冲突解决核心：业务文件保留 main 最终版，测试分支专属配置/删除保留 yunjin 版本。
